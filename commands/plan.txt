Load the skill `skills_tool:load planning-and-task-breakdown` and follow its instructions.

Read the existing spec (SPEC.md or equivalent) and the relevant codebase sections. Then:

1. Enter plan mode — read only, no code changes
2. Identify the dependency graph between components
3. Slice work vertically (one complete path per task, not horizontal layers)
4. Write tasks with acceptance criteria and verification steps
5. Add checkpoints between phases
6. Present the plan for human review

Save the plan to `tasks/plan.md` and task list to `tasks/todo.md`.

---

⚠️ **MANDATORY** — After the plan is written, you MUST call `lifecycle:init` to activate the lifecycle runtime layer:

```
lifecycle:init goal="<plan goal>" slug="<project-slug>"
```

If `lifecycle:init` fails, resolve the error before proceeding. Do not skip this step.

If the planning tool is not installed (no `plan` tool available), skip this step gracefully.
{raw}

## Exit Criteria (must ALL be true before proceeding to /build)
- [ ] tasks/plan.md and tasks/todo.md exist at project root
- [ ] Dependency graph is documented
- [ ] Each task has acceptance criteria and verification steps
- [ ] User has reviewed and approved the plan

## Anti-Rationalization Table
| Rationalization | Reality |
|---|---|
| "The plan is clear in my head" | Heads forget. Plans persist. Write it down. |
| "We'll figure out dependencies as we go" | Surprise dependencies cause rework. Map them upfront. |
| "Planning is overhead" | 10 minutes of planning saves 2 hours of rework. |

## Red Flags
- No dependency analysis
- Tasks are horizontal layers instead of vertical slices
- No verification steps per task
- User hasn't reviewed the plan
